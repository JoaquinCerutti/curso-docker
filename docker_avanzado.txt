MultiStage

# MULTISTAGE BUILD EN DOCKER

# Es una técnica que permite usar múltiples "stages" (etapas) dentro de un mismo Dockerfile.
# Cada stage puede tener su propia imagen base y propósito (build, test, runtime, etc).

# IDEA CLAVE:
# - En la etapa de build instalás TODO (compiladores, dependencias pesadas, etc)
# - En la etapa final copiás SOLO lo necesario (binarios o app ya compilada)
# => Resultado: imágenes mucho más livianas y seguras.

Sin multistage:
Imagen final = build tools + código + basura

Con multistage:
Imagen final = SOLO app lista para correr

# --------------------------------------------------
# ¿POR QUÉ OPTIMIZA TAMAÑO?
# --------------------------------------------------
# Porque descartás:
# - Compiladores (gcc, dotnet sdk, node build tools, etc)
# - Dependencias de build
# - Archivos temporales
# - Código fuente innecesario

# Solo queda lo que realmente se ejecuta.

# --------------------------------------------------
# EJEMPLO SIMPLE (NODE)
# --------------------------------------------------

# Stage 1: build
FROM node:18 AS build

WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# Stage 2: runtime (imagen liviana)
FROM node:18-alpine

WORKDIR /app

# Copiamos SOLO lo necesario del stage anterior
COPY --from=build /app/dist ./dist
COPY --from=build /app/node_modules ./node_modules
COPY package*.json ./

CMD ["node", "dist/index.js"]

Docker y permisos
Todas las imagenes que uno crea, se crean con el usuario root



6 prácticas recomendadas para sacar imagenes a producción:

1. Usar imágenes oficiales.
2. Evitar usar latest, puede presentar detalles/errores.
3. Imagenes alpine son de reducido tamaño --> muy buena práctica.
4. Agregar multistage a las imagenes.
5. Eliminar constantemente archivos no utilizados en docker (prune).
6. Crear usuarios dentro del archivo docker, evitar el usuario root por defecto.






Subir imagenes a microsoft azure
No es un dockerhub, es privado

Creamos un Container Registry "*name*.azurecr.io" por ejemplo

En consola:

az login
az account show
docker --version
docker ps
az acr login -n registryjoaquin

 docker build -t *name*.azurecr.io/sitioweb:latest .
 docker images  
 docker push *name*.azurecr.io/sitioweb:latest


*** Tener la imagen en un registro de contenedores de Azure no la hace exclusiva de Azure, si privado ***


Azure Container Registry (ACR) es un registro privado de imágenes de contenedores en la nube de Azure.

En términos simples:

Es como un Docker Hub, pero privado y dentro de tu cuenta de Azure.
Ahí podés guardar, versionar y gestionar imágenes Docker de tus aplicaciones.
Luego esas imágenes se usan para desplegar contenedores en servicios como:
Kubernetes (AKS)
Azure Container Instances
